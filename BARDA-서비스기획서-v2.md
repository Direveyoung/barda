# BARDA (바르다) — 서비스 기획서 v2.2
### "바르게 바르다" | 화장품 루틴/성분 충돌 체크 + 루틴 커뮤니티 서비스
**최종 업데이트: 2026.02.16**

---

## 0. 서비스 개요

| 항목 | 내용 |
|---|---|
| **서비스명** | BARDA (바르다) |
| **슬로건** | 바르게 바르다 |
| **한 줄 정의** | 내가 가진 화장품을 검색·입력하면, AM/PM 루틴 순서 + 위험 조합 경고 + 7일 캘린더를 한 번에 받는 서비스 |
| **네이밍 의미** | "바르다(apply)" + "바르다(right)"의 중의적 표현. 영문 BARDA는 K-뷰티 글로벌 확장에 그대로 사용 가능 |
| **타겟** | 한국 2030 여성 (스킨케어 제품 평균 7~12개 보유, 조합법을 모르는 사용자) |
| **핵심 가치** | 개인화된 루틴 최적화 + 루틴 중심 커뮤니티 (의학적 판정 ❌ → 뷰티 가이드 ✅) |
| **서비스 형태** | 모바일 웹앱 (PWA) → 네이티브 앱 확장 |

### 앱 전체 구조 (모바일 웹앱)

```
BARDA 앱
┌─ 🏠 홈: 내 루틴 체크 (핵심 도구)
│   ├─ 피부타입·고민 입력
│   ├─ 제품 검색/등록
│   └─ AM/PM 루틴 리포트 + 점수 + 캘린더
│
├─ 📱 피드: 루틴 공유 (커뮤니티 핵심)
│   ├─ 피부타입·고민별 필터
│   ├─ 인기 루틴 랭킹 (주간/월간)
│   └─ 루틴 카드 좋아요·저장·댓글
│
├─ 💬 Q&A: 루틴 질문/답변 (정보)
│   ├─ 사용자 질문 + 커뮤니티 답변
│   └─ BARDA 공식 가이드 자동 보조
│
├─ 🔥 챌린지: 7일 루틴 일지 (참여·리텐션)
│   ├─ BARDA 캘린더 기반 따라하기
│   └─ #바르다7일챌린지 SNS 연동
│
├─ 📖 성분사전: 성분별 가이드 (정보 인프라)
│   ├─ 핵심 성분 30종 가이드
│   └─ 충돌 룰 연동 ("왜?" 클릭 → 성분사전)
│
└─ 👤 마이페이지: 내 루틴 히스토리
    ├─ 과거 분석 결과
    ├─ 저장한 루틴
    └─ 챌린지 기록
```

## 1. 시장 분석

### 1-1. 화해(Hwahae) 벤치마킹

**화해가 잘하는 것**
- 33만개+ 제품 성분 DB (국내외 화장품 성분 분석 데이터)
- 900만+ 리뷰 (내돈내산 실사용자 리뷰)
- 피부타입별 필터링 (건성/지성/복합성/민감성별 리뷰 열람)
- 성분 사전 (기능별 성분 분류, 즐겨찾기 성분 포함/제외 필터)
- 마이 성분 기능 (자신에게 맞는/안 맞는 성분 저장)
- 리뷰 토픽 (AI가 600만+ 리뷰를 분석해 키워드로 정리)
- 1,000만+ 다운로드, 2024 매출 838억원 (연결 기준)

**화해가 안 하는 것 = BARDA의 기회 ⭐**

| 화해에 없는 것 | BARDA의 답 |
|---|---|
| "내 제품 전체"를 한눈에 보는 루틴 뷰 | 보유 제품 검색/입력 → AM/PM 루틴 자동 생성 |
| 제품 간 충돌/조합 경고 | "레티놀+AHA 같은 날" 경고 카드 |
| 사용 순서 가이드 | 제형 기반 자동 순서 배치 |
| 요일별 루틴 캘린더 | 레티놀날/각질날/보습날 자동 분배 |
| "욕심 루틴" 진단 | 액티브 과다 사용 경고 |
| AM/PM별 맞춤 가이드 | 피부타입×시간대별 루틴 TIP |

> **핵심 차별점**: 화해는 "제품 1개의 성분 분석"에 강하지만, **"내가 가진 제품 5~10개를 어떻게 조합해 쓸까"**에 대한 답은 없음. BARDA는 이 빈틈을 정확히 공략.

**화해의 약점에서 배울 점**
- EWG 성분 유해도 논란: 성분 위험성 판단이 과학적 근거 부족하다는 비판 지속
- 정보 경유지 역할에 머무름: 정보만 보고 올리브영/쿠팡에서 구매
- 커머스 전환 어려움: 2024년 영업손실 28억, 완전자본잠식 지속
- **교훈**: BARDA는 "성분 판정"이 아닌 **"루틴 최적화 도구"**로 포지셔닝. 의학적 주장 절대 회피.

### 1-2. 글로벌 경쟁 서비스 분석

| 서비스 | 국가 | 핵심 기능 | 강점 | 약점 |
|---|---|---|---|---|
| **SkinSort** | 미국 | 제품 스캔 → 루틴 빌더 → 충돌 경고 → 커뮤니티 루틴 공유 | DB 풍부, 듀프 찾기 | 한국 제품 없음, 한국어 미지원 |
| **Skin Bliss** | 미국 | AI 얼굴 스캔 → 루틴 빌더 → 진행 트래킹 | 선반 사진 제품 인식, 날씨 기반 루틴 | 구독 비쌈, 한국 미진출 |
| **Skincare Routine App** | 영국 | The Ordinary 중심 → 순서/충돌 자동 배치 → 사용 추적 | 원타임 결제, 요일별 스케줄링 | The Ordinary 편중, 한국 제품 없음 |

> **결론: 한국 시장에 "루틴 빌더 + 충돌 체크"를 제대로 하는 서비스가 없다.** K-뷰티 시장을 위한 명확한 빈틈.

### 1-3. BARDA 포지셔닝

| | 화해 | BARDA |
|---|---|---|
| **관점** | 제품 중심 (이 제품 괜찮아?) | 루틴 중심 (이렇게 쓰면 돼?) |
| **입력** | 제품 1개 검색 | 내 제품 전체 검색/입력 |
| **출력** | 성분 분석 + 리뷰 | AM/PM 루틴 + 충돌 경고 + 캘린더 + TIP |
| **수익** | 광고 + 커머스 수수료 | 리포트 판매 + 구독 |
| **관계** | 경쟁 아님, 보완 관계 | 화해에서 정보 → BARDA에서 조합 |

### 1-4. 데이터 플라이휠 (핵심 성장 전략)

```
제품 DB 확장 → 검색 히트율↑ → 사용자 만족도↑ → 리포트 공유(바이럴)
     ↑                                                    ↓
  DB 정교해짐 ← 사용자 데이터 축적 ← 신규 유입 ← 커뮤니티 자랑/공유
```

> **DB가 이 서비스의 진짜 해자(moat)**. 제품 DB가 두꺼워질수록 UX가 좋아지고, 사용자가 늘면 데이터가 쌓이고, 커뮤니티·바이럴·리텐션이 자동으로 따라오는 구조.

## 2. 서비스 설계

### 2-1. 사용자 플로우

```
[STEP 1] 피부타입 선택
   건성 / 지성 / 복합성 / 민감성

[STEP 2] 피부 고민 선택 (최대 2개)
   트러블 / 잡티·색소 / 탄력·주름 / 각질 / 보습 / 모공

[STEP 3] 보유 제품 등록
   ┌─ [우선] 제품명/브랜드 검색 → 자동완성 → 카테고리 자동 매핑
   └─ [대안] 검색에 없으면 → 수동 카테고리 선택 + 제품명 입력
   + 사용빈도 선택 (매일 / 주3~4회 / 주1~2회 / 가끔)

[STEP 4] 결과 리포트 생성
   → 무료: 간단 루틴 요약 (제품 3개까지)
   → 유료(9,900원): 전체 리포트
```

**카테고리 분류:**
- 기초: 클렌저 / 오일클렌저 / 토너 / 토너패드 / 에센스 / 세럼 / 크림 / 로션 / 오일 / 아이크림 / 미스트
- 기능성: 비타민C / 레티놀 / AHA제품 / BHA제품 / PHA제품 / 나이아신아마이드 / 펩타이드
- 트러블: 벤조일퍼옥사이드 / 트러블스팟 / 시카제품
- 기타: 선크림 / 시트마스크 / 스크럽/필링 / 수면팩

### 2-2. 제품 자동검색 시스템

**MVP 제품 DB 구조:**
```json
{
  "id": "p001",
  "name": "이니스프리 그린티 클렌징폼",
  "brand": "이니스프리",
  "categoryId": "cleanser",
  "tags": ["기초", "클렌저"]
}
```

**검색 로직:**
- 제품명, 브랜드명 모두 검색 대상
- 한 글자부터 자동완성 시작
- 최대 8개 결과 표시
- 클릭 시 카테고리 자동 매핑 → 즉시 제품 목록에 추가
- 검색에 없으면 "직접 입력" 폴백 제공

**초기 DB 규모:**
- MVP: 인기 제품 80~100개 (프로토타입에 구현 완료)
- Phase 2: 200~500개로 확장 (검색 히트율 80%+ 목표)
- Phase 3: 사용자 직접 입력 데이터를 기반으로 자동 확장

**DB 확장 우선순위 (브랜드):**
라운드랩, 토리든, 아누아, 코스알엑스, 이니스프리, 넘버즈인, 에스트라, 마녀공장, 닥터지, 클레어스, 구달, 메디힐, VT, 라네즈, 벨리프, 키엘, SK-II, 라로슈포제, 디오디너리, 폴라초이스, 비오레, 일리윤 등

### 2-3. 리포트 구성

**1. 루틴 점수 (100점 만점)**
- 순서 정확도 / 충돌 없음 / 필수 단계 충족 / 균형도 → 종합 점수
- 점수별 멘트: 90+ "완벽한 루틴!" / 80+ "좋은 루틴" / 60+ "개선하면 더 좋아져요" / 60 미만 "점검 필요"

**2. 빠진 필수 단계 알림**
- 선크림 미사용 → 🔴 최우선 경고
- 보습 크림/로션 없음 → 🟡 권장 경고
- 클렌저 없음 → 🟡 권장 경고
- 레티놀 사용 중 선크림 없음 → 🔴 필수 경고

**3. 아침 루틴 (AM Routine) — "가볍게, 보호 중심"**
- 제품별 사용 순서 시각화 (번호 + 제품명 + 카테고리)
- AM 전용 TIP (피부타입·고민별 맞춤):
  - 선크림 사용법 (양, 타이밍)
  - 비타민C + 선크림 시너지
  - 지성피부 아침 가벼운 보습 가이드
  - 화장 밀림 방지 팁

**4. 저녁 루틴 (PM Routine) — "집중 케어, 회복 중심"**
- 제품별 사용 순서 시각화
- PM 전용 TIP (피부타입·고민별 맞춤):
  - 이중세안 방법
  - 레티놀 사용 타이밍/주의사항
  - 각질케어 제품 사용법
  - 수분 봉인 (크림/오일/수면팩)
  - 건성피부 저녁 집중 보습 가이드
  - 민감피부 장벽 회복 팁

> **AM/PM 차별화 핵심**: 제품이 적더라도(3~5개) 시간대별 맞춤 TIP으로 "분석받은 가치"를 체감하게 함. 아침은 보호·가벼움, 저녁은 케어·회복으로 톤 분리.

**5. 충돌 경고 카드 (3~7개)**
- 경고 수준 (🔴높음 / 🟡중간 / 🟢낮음) + 배지
- 관련 제품명 표시
- 왜 문제인지 한 줄 설명
- "이렇게 바꿔보세요" 대안
- 피부타입별 심각도 조정

**6. 7일 루틴 캘린더 (탭 전환)**
- 월~일 요일별 AM/PM 사용 제품 배치
- 레티놀날(화,목,토) / 각질케어날(수,일) / 기본 루틴 자동 분배
- 각 날짜 이모지 + 라벨로 시각화
- 액티브 제품 없으면: "제품을 추가하면 요일별 캘린더가 생겨요" (자연스러운 업셀링)

## 3. 룰 엔진 (카테고리 기반 50개 룰)

### A. 루틴 순서 룰 (10개)

| # | 룰 | 설명 |
|---|---|---|
| A01 | AM 순서 | 클렌저 → 토너 → 에센스 → 세럼 → 아이크림 → 크림/로션 → 선크림 |
| A02 | PM 순서 | 클렌징오일/밤 → 폼클렌저 → 토너 → 에센스 → 세럼 → 아이크림 → 크림 |
| A03 | 제형 순서 원칙 | 묽은 제형 → 걸쭉한 제형 순서 (워터 → 에센스 → 세럼 → 크림 → 오일) |
| A04 | 더블클렌징 PM | 저녁에 메이크업/선크림 사용 시 오일클렌저 + 폼클렌저 이중세안 권장 |
| A05 | 선크림 AM 필수 | 아침 루틴 마지막에 선크림 없으면 → ⚠️ 경고 (만족도 최고 룰) |
| A06 | 토너패드 위치 | 토너패드(각질케어 성분 포함)는 토너 단계에서 사용 |
| A07 | 시트마스크 위치 | 세럼 후, 크림 전 (에센스 흡수 극대화) |
| A08 | 스팟케어 위치 | 크림 바른 후 해당 부위에만 마지막 적용 |
| A09 | 오일 위치 | 크림 후 마지막 단계 (봉인 역할) 또는 크림에 섞어 사용 |
| A10 | 미스트 위치 | 토너 후 또는 메이크업 위에 세팅용으로 사용 |

### B. 성분 충돌 룰 (15개)

| # | 조합 | 경고 수준 | 권장 |
|---|---|---|---|
| B01 | 레티노이드 + AHA(글리콜산) | 🔴 높음 | 같은 날 사용 금지, 번갈아 사용 |
| B02 | 레티노이드 + BHA(살리실산) | 🟡 중간 | 민감피부는 분리, 지성은 주의하며 가능 |
| B03 | 레티노이드 + 벤조일퍼옥사이드 | 🔴 높음 | 동시 사용 시 레티놀 분해, AM/PM 분리 |
| B04 | 레티노이드 + 비타민C(고농도 L-AA) | 🟡 중간 | pH 차이로 효능 감소, AM(비타민C)/PM(레티놀) 분리 권장 |

  id TEXT PRIMARY KEY,              -- "cleanser"

);

CREATE TABLE user_routines (

  skin_type TEXT NOT NULL,
  concerns TEXT[],
  products JSONB NOT NULL,          -- 등록한 제품 목록
  result JSONB NOT NULL,            -- 분석 결과 전체
  score INTEGER,
  is_paid BOOLEAN DEFAULT false,

);

## 5. 제품 DB 구축 전략

> **DB가 이 서비스의 핵심 자산이자 해자(moat)**. 검색해서 안 나오면 이탈하고, 나오면 만족한다. 검색 히트율이 곧 사용자 만족도.

| 올리브영 카테고리별 베스트셀러 Top 20 | 가장 많이 팔리는 제품 = 가장 많이 검색할 제품 | ~200개 |

**주요 필드 설명:**

| 필드 | 용도 | 예시 |
|---|---|---|
| `name_normalized` | 검색 최적화 (공백·특수문자 제거) | "라운드랩독도토너" |

| `tags` | 검색·필터용 태그 | ["수분", "저자극"] |
| `source` | 데이터 출처 추적 | "manual_v1", "api_mfds", "user_promoted" |
| `verified` | 관리자 검증 여부 | true/false |
| `search_count` | 검색 인기도 (자동 증가) | 0 → 누적 |

**구축 작업 효율화:**
- 제품당 입력 시간: ~2~3분 (제품명, 브랜드, 카테고리, 핵심 성분 확인)
- 200개 = 약 8~10시간 수작업
- **Claude Code 반자동화**: 올리브영 베스트 목록 텍스트 → JSON 변환 스크립트로 절반 이하로 단축 가능

### 5-3. 3단계: 사용자 기반 자동 학습 업데이트 (Month 2+)

**목표**: 사용자가 쓸수록 DB가 자동으로 풍성해지는 자기강화 루프 구축

#### ① 사용자 직접 입력 → DB 후보 자동 수집

검색에서 제품을 못 찾아 "직접 입력"으로 제품명 + 카테고리를 입력하면, 이 데이터가 자동 수집됨.

```sql
-- 사용자 입력 제품 후보 테이블
CREATE TABLE product_candidates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  raw_name TEXT NOT NULL,              -- 사용자가 입력한 원본 제품명
  name_normalized TEXT NOT NULL,       -- 정규화된 이름 (소문자, 공백제거)
  brand_guess TEXT,                    -- 사용자 입력 또는 자동 추출 브랜드
  category_id TEXT NOT NULL,           -- 사용자가 선택한 카테고리
  submit_count INTEGER DEFAULT 1,      -- 같은 제품 입력 횟수 (중복 시 +1)
  unique_users INTEGER DEFAULT 1,      -- 고유 사용자 수
  last_submitted_at TIMESTAMPTZ,
  is_promoted BOOLEAN DEFAULT false,   -- 정식 DB 승격 여부
  promoted_at TIMESTAMPTZ,
  promoted_product_id TEXT,            -- 승격된 경우 정식 product ID
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 중복 감지: name_normalized 기준으로 UPSERT
-- 새 입력 시 submit_count +1, unique_users 갱신
```

**자동 승격 조건:**
```
IF submit_count >= 3 AND unique_users >= 2:
    → product_candidates.is_promoted 후보 플래그 ON
    → 관리자 대시보드에 "승격 대기" 알림
    → 관리자 확인 (이름 정규화 + 카테고리 검증 + 핵심 성분 추가)
    → products 테이블에 정식 등록 (source: "user_promoted")
    → 다음 사용자부터 검색 가능
```

**Phase 3 자동화:**
```
IF submit_count >= 5 AND unique_users >= 3 AND category 일치율 >= 80%:
    → 자동 승격 (관리자 확인 없이)
    → 주간 자동 승격 리포트 생성 → 관리자 사후 검증
```

#### ② 검색 로그 분석 → DB 갭 발견

```sql
-- 검색 로그 테이블
CREATE TABLE search_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID,                        -- 로그인 사용자 (없으면 NULL)
  query TEXT NOT NULL,                 -- 검색어
  query_normalized TEXT NOT NULL,      -- 정규화 검색어
  results_count INTEGER NOT NULL,      -- 검색 결과 수
  selected_product_id TEXT,            -- 사용자가 선택한 제품 (있으면)
  fell_through BOOLEAN DEFAULT false,  -- 검색 실패 → 직접입력으로 이동했는지
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**핵심 지표 모니터링:**

| 지표 | 산출 방법 | 목표 | 액션 |
|---|---|---|---|
| **검색 히트율** | (results_count > 0 비율) | 80%+ | 70% 미만 시 DB 확장 필요 신호 |
| **선택 전환율** | (selected 있음 / 검색 수) | 60%+ | 낮으면 검색 결과 품질 이슈 |
| **이탈율** | (fell_through 비율) | 15% 이하 | 높으면 DB 갭 큼 |
| **검색 미스 Top 20** | fell_through=true인 query 빈도순 | 주간 추출 | 우선 DB 추가 대상 |

**주간 DB 갭 리포트 자동 생성:**
```sql
-- 주간 검색 미스 Top 20 쿼리
SELECT query_normalized, COUNT(*) as miss_count, 
       COUNT(DISTINCT user_id) as unique_users
FROM search_logs 
WHERE fell_through = true 
  AND created_at > NOW() - INTERVAL '7 days'
GROUP BY query_normalized
ORDER BY miss_count DESC
LIMIT 20;
```

## 9. 커뮤니티 설계

> **핵심 원칙**: 화해 = "이 제품 어때?" (제품 리뷰). BARDA = **"이 조합 어때?" (루틴 공유)**. 제품 리뷰는 레드오션이지만, 루틴 공유 커뮤니티는 아직 아무도 안 하고 있음.

### 8-1. 루틴 공유 피드 (커뮤니티 핵심 — MVP 포함)

BARDA에서 분석받은 루틴 카드 자체가 콘텐츠가 되는 구조.

**루틴 카드 구성:**
- 루틴 점수 (예: 92점)
- 피부타입 + 고민 태그 자동 부착
- AM/PM 제품 목록 요약
- 충돌 경고 수 (예: "충돌 0건 ✅")
- 한 줄 코멘트 (사용자 작성)

**피드 기능:**
- 피부타입·고민별 필터링 ("나와 같은 건성+보습 루틴만 보기")
- 좋아요 / 저장 / 댓글
- 인기 루틴 랭킹 (주간/월간)
- "이 루틴 따라하기" → 내 루틴 체크에 제품 자동 입력

**화해와의 차별점:**

| | 화해 커뮤니티 | BARDA 피드 |
|---|---|---|
| 콘텐츠 단위 | 제품 1개 리뷰 | 루틴 전체 (5~10개 조합) |
| 핵심 정보 | "이 제품 좋아/싫어" | "이렇게 조합하니까 좋았어" |
| 데이터 가치 | 제품별 평점 | **조합 패턴 데이터** |
| 탐색 방식 | 제품 검색 → 리뷰 읽기 | 피부타입 필터 → 루틴 참고 |

### 8-2. 루틴 Q&A (정보 채널 — MVP 포함)

**구조:**
- 사용자가 루틴/성분 관련 질문 등록
- 다른 사용자가 답변 + 좋아요로 베스트 답변 선정
- **BARDA 공식 가이드 자동 보조**: 질문에 특정 성분/조합 키워드가 포함되면 룰 엔진 기반 공식 답변이 자동으로 표시

**Q&A 예시:**
```
Q: "레티놀 입문하려는데 어떤 제품부터 시작할까요? (민감성)"
├─ 🏷️ BARDA 가이드: "민감피부는 주 2~3회, 저농도부터 시작하세요. 
│   같은 날 AHA/BHA 사용은 피해주세요." (룰 C03, B01 기반)
├─ 💬 사용자A: "코스알엑스 레티놀 0.1로 시작했는데 순해서 좋았어요"
└─ 💬 사용자B: "이니스프리 레티놀 시카는 시카 성분이 같이 들어있어서..."
```

**포지셔닝**: 화해 Q&A가 "제품 추천"이라면, BARDA Q&A는 **"조합/루틴 전문 지식인"**.

### 8-3. 7일 루틴 챌린지 (참여·리텐션 — Phase 2)

BARDA에서 받은 7일 캘린더를 실제로 따라하며 기록하는 기능.

**챌린지 플로우:**
```
루틴 분석 완료 → "7일 챌린지 시작하기" 버튼
→ Day 1~7: 매일 간단 체크인
   - 오늘의 루틴 완료 ✅
   - 피부 컨디션 (😊좋음 / 😐보통 / 😣자극)
   - 한 줄 메모 (선택)
   - 셀피 기록 (선택)
→ 7일 완료: 챌린지 완료 배지 + 결과 요약 카드
```

**바이럴 설계:**
- #바르다7일챌린지 해시태그로 SNS 공유
- 챌린지 완료 카드: "7일간 루틴 점수 78→91점으로 개선!" 같은 비포/애프터
- 챌린지 참여자 피드 별도 탭

**리텐션 효과:**
- 7일 연속 방문 유도 (DAU 핵심 지표)
- 실제 사용 후기가 자연스럽게 UGC로 축적
- 챌린지 완료 후 → "제품 바꿔볼까?" → 재분석 유도 (구독 전환 포인트)

### 8-4. 성분사전 (정보 인프라 — MVP 포함)

커뮤니티의 신뢰도를 만드는 기반 정보 레이어.

**초기 구성 (핵심 성분 30종):**

| 카테고리 | 성분 |
|---|---|
| 각질케어 | AHA(글리콜산), BHA(살리실산), PHA(글루코노락톤) |
| 안티에이징 | 레티놀, 레틴알데히드, 바쿠치올, 펩타이드, 아데노신 |
| 미백·톤업 | 비타민C(L-AA), 나이아신아마이드, 알파아르부틴, 트라넥사믹산, 글루타치온 |
| 보습 | 히알루론산, 세라마이드, 스쿠알란, 판테놀, 글리세린 |
| 진정 | 시카(센텔라), 알로에, 마데카소사이드, 알란토인 |
| 트러블 | 벤조일퍼옥사이드, 티트리, 아젤라인산 |
| 기타 | 구리펩타이드, 효소(파파인) |

**각 성분 페이지 구성:**
- 한 줄 설명 + 효과
- 추천 피부타입
- ⚠️ 주의 조합 (충돌 룰 연동)
- ✅ 추천 조합 (시너지)
- 대표 제품 (BARDA DB 연동)
- 관련 Q&A 자동 링크

**사용자 여정 연동:**
```
충돌 경고 카드 "레티놀 × AHA" → "왜?" 클릭 → 성분사전 레티놀 페이지
→ 주의 조합 섹션에서 상세 설명 확인 → 관련 Q&A 탐색
```

### 8-5. 커뮤니티 데이터 플라이휠

```
루틴 공유 피드 데이터 축적
    ↓
"건성+보습 Top 10 인기 루틴" 자동 생성
    ↓
피부타입별 인기 조합 패턴 분석 가능
    ↓
"비슷한 피부타입 사용자들이 이렇게 씁니다" AI 추천 (Phase 3)
    ↓
추천 정확도↑ → 서비스 가치↑ → 더 많은 사용자 유입
    ↓
더 많은 루틴 데이터 축적... (반복)
```

> **제품 DB 플라이휠 + 커뮤니티 데이터 플라이휠** 이 동시에 돌아가면 경쟁자가 따라올 수 없는 해자가 만들어짐.

### 8-6. 커뮤니티 운영 정책

**콘텐츠 가이드라인:**
- ✅ 허용: 루틴 공유, 제품 조합 후기, 피부 변화 기록, 성분 질문
- ❌ 금지: 의학적 진단/처방 주장, 특정 제품 악의적 비방, 광고성 글, 개인정보 노출
- BARDA 공식 답변에는 항상 "일반 뷰티 가이드" 고지 포함

**모더레이션 우선순위 (MVP):**
- Phase 1: 신고 기반 사후 관리 + 금칙어 자동 필터
- Phase 2: AI 기반 부적절 콘텐츠 자동 감지
- Phase 3: 커뮤니티 매니저 + 파워유저 등급제

### 8-7. 커뮤니티 DB 설계

```sql
-- 루틴 공유 게시글
CREATE TABLE routine_posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users,
  routine_id UUID REFERENCES user_routines, -- 분석 결과 연동
  comment TEXT,                              -- 한 줄 코멘트
  skin_type TEXT NOT NULL,
  concerns TEXT[],
  score INTEGER,
  product_count INTEGER,
  conflict_count INTEGER,
  likes_count INTEGER DEFAULT 0,
  saves_count INTEGER DEFAULT 0,
  is_public BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 좋아요
CREATE TABLE post_likes (
  user_id UUID REFERENCES auth.users,
  post_id UUID REFERENCES routine_posts,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, post_id)
);

-- 저장 (북마크)
CREATE TABLE post_saves (
  user_id UUID REFERENCES auth.users,
  post_id UUID REFERENCES routine_posts,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, post_id)
);

-- 댓글
CREATE TABLE post_comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES routine_posts,
  user_id UUID REFERENCES auth.users,
  content TEXT NOT NULL,
  likes_count INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Q&A 질문
CREATE TABLE qa_questions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  tags TEXT[],                               -- ["레티놀", "민감성", "입문"]
  auto_guide JSONB,                          -- BARDA 룰엔진 자동 답변
  views_count INTEGER DEFAULT 0,
  answers_count INTEGER DEFAULT 0,
  is_resolved BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Q&A 답변
CREATE TABLE qa_answers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  question_id UUID REFERENCES qa_questions,
  user_id UUID REFERENCES auth.users,
  content TEXT NOT NULL,
  is_best BOOLEAN DEFAULT false,
  likes_count INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 챌린지 참여
CREATE TABLE challenge_entries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users,
  routine_id UUID REFERENCES user_routines,
  day_number INTEGER NOT NULL,               -- 1~7
  is_completed BOOLEAN DEFAULT false,
  skin_condition TEXT,                        -- "good" | "normal" | "irritated"
  memo TEXT,
  photo_url TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 성분사전
CREATE TABLE ingredients (
  id TEXT PRIMARY KEY,                       -- "retinol"
  name_ko TEXT NOT NULL,                     -- "레티놀"
  name_en TEXT,                              -- "Retinol"
  category TEXT NOT NULL,                    -- "안티에이징"
  description TEXT NOT NULL,                 -- 한 줄 설명
  effects TEXT[],                            -- ["주름개선", "피부재생", "콜라겐 촉진"]
  recommended_skin_types TEXT[],             -- ["all"]
  caution_skin_types TEXT[],                 -- ["sensitive"]
  conflict_rules TEXT[],                     -- ["B01", "B03", "B04", "B10"]
  synergy_ingredients TEXT[],                -- ["niacinamide", "hyaluronic_acid"]
  usage_tip TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```
